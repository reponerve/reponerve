# RepoNerve competitive landscape - 2026-06-30

Status: Draft

Purpose: Capture competitive research for RepoNerve and convert the findings into issue-ready backlog items.

> Vision anchor: RepoNerve reduces the cost of software understanding through knowledge preservation - local, evidence-backed memory that survives beyond individual contributors and works for humans and AI agents.

## Research scope

Primary competitors and adjacent tools reviewed:

| Category | Tools | Competitive signals |
| --- | --- | --- |
| Codebase-aware review | Greptile, CodeRabbit, Augment Code Review | Full-repo graph context, adaptive rules, PR summaries, issue/ticket context, test generation, IDE handoff for fixes |
| Enterprise code intelligence | Sourcegraph Cody, Sourcegraph Amp, Augment Context Engine | Multi-repo context, access controls, search plus chat, context filters, hosted and self-hosted deployment |
| Local / open context engines | Aider repo-map, Serena, CodeGraph, Cortex, RepoRelay, MemCP, RepoMem | MCP-native or CLI-first retrieval, symbol graphs, PageRank-style context selection, local storage, LSP/tree-sitter support |
| Developer memory | Pieces.app, Cursor memories/rules/skills, Claude/agent rule files | Persistent project and workflow memory, OS-level capture, privacy controls, reusable procedures |
| Spec and documentation surfaces | AWS Kiro, DeepWiki/Devin Wiki | Spec-driven planning, steering files, hooks, generated source-cited repo wikis and diagrams |

## Ecosystem trends

1. **MCP is becoming the distribution layer for context.** Augment, DeepWiki, Serena, Pieces, Cursor, and many smaller tools expose context through MCP-compatible clients.
2. **Graph-aware context is table stakes.** Aider uses tree-sitter and PageRank-style repo maps; Greptile and CodeRabbit market code graphs for PR review; Serena exposes LSP-backed symbol operations.
3. **Competitors sell proof, not just architecture.** Augment publishes agent-quality claims; Aider documents token-budgeted repo maps; code review vendors emphasize defect-catch rates and review speed.
4. **Review and validation are the fastest-growing surfaces.** Greptile, CodeRabbit, and Augment position context as the central validation layer for code generated by other agents.
5. **Rules and specs are becoming repository artifacts.** Kiro steering files, Cursor rules, Claude/AGENTS instructions, Greptile rules, CodeRabbit knowledge-base config, and Devin wiki config all encode project truth.
6. **Local-first privacy is a growth wedge.** Pieces, Cortex, Aider, Serena, CodeGraph, and RepoMem all use local or repo-native positioning against SaaS review and enterprise search platforms.
7. **Generated, browsable understanding is expected.** DeepWiki turns repositories into source-cited wiki pages and diagrams; RepoNerve's local `explore` UI is aligned but not yet packaged as an onboarding artifact.

## RepoNerve positioning constraints

RepoNerve already ships local SQLite memory, deterministic code intelligence for Go plus 19 tree-sitter languages, repository-code linking, feature understanding, bounded context packs, session memory, 49 MCP tools, `reuse-check`, `ship-check`, `pr-context`, `doctor`, scoped scan, npm distribution, and a capped local `explore` UI.

Current release-line non-goals remain important:

- Do not make semantic or hybrid embedding search the primary authority.
- Do not add mandatory SaaS to the core product.
- Do not add autonomous code modification, commit, or deployment.
- Do not add Sourcegraph-scale cross-repo federation without a new RFC.
- Do not expand into a full enterprise graph explorer without a new RFC.

### 2. Publish a reproducible context-quality and token-savings benchmark suite

**Type:** feature

**Priority:** P0

**Proposed labels:** `type:feature`, `priority:p0`, `enhancement`

**Problem**

RepoNerve's strongest claim is reduced exploration and token cost, but the project lacks a reproducible benchmark that proves context quality and token savings against obvious baselines.

**Competitive evidence**

Augment markets Context Engine MCP with agent-quality and tool-call reduction claims. Aider documents token-budgeted repo maps. Greptile and CodeRabbit sell full-codebase context as a measurable review-quality advantage.

**RepoNerve alignment**

This directly supports the token economics and universal-understanding positioning. It should remain deterministic and evidence-backed, using RepoNerve outputs rather than model-judged conclusions as primary evidence.

**Acceptance criteria**

- Add a benchmark plan and runnable harness comparing RepoNerve against at least:
  - raw repository grep/read exploration
  - Aider-style structural repo-map baseline, where feasible
  - a no-memory baseline for `ask`, `plan`, and `review` tasks
- Measure at minimum:
  - tokens or bytes emitted to the agent
  - number of files or entities surfaced
  - evidence coverage for the expected answer
  - latency for scan and query
- Include at least one fixture repository beyond RepoNerve itself.
- Publish results in `docs/product/` with methodology, caveats, and reproducible commands.
- Do not use embeddings or remote model calls as required benchmark infrastructure.

**RFC needed:** No for a benchmark harness; yes if it introduces new product APIs or telemetry storage.

### 3. Add a public competitive positioning page for "Software Understanding Infrastructure"

**Type:** suggestion

**Priority:** P0

**Proposed labels:** `type:suggestion`, `priority:p0`, `documentation`

**Problem**

RepoNerve's differentiation can be confused with code graph retrieval, semantic search, AI review, or agent memory. The product needs a concise comparison page explaining when RepoNerve complements or differs from Greptile, Sourcegraph Cody, Augment, Pieces, Aider, Serena, DeepWiki, Kiro, and CodeRabbit.

**Competitive evidence**

Competitors have crisp category narratives: Greptile and CodeRabbit are validation layers, Sourcegraph Cody is enterprise code understanding, Augment is a context engine for any MCP agent, Pieces is OS-level memory, Aider is terminal pair-programming with a repo map, Serena is an IDE-like symbol toolkit, DeepWiki is generated repository documentation, and Kiro is spec-driven development.

**RepoNerve alignment**

RepoNerve's own vision says the differentiator is knowledge preservation and software understanding, not code retrieval alone. A positioning page reduces adoption friction and helps users compose RepoNerve with adjacent tools.

**Acceptance criteria**

- Add or update a product page with:
  - a competitor matrix
  - direct and indirect competitor categories
  - "RepoNerve complements this tool when..." guidance
  - explicit non-goals and RFC boundaries
  - local-first and evidence-backed differentiators
- Include source links for competitor claims.
- Keep claims factual; avoid unverified win/loss assertions.

**RFC needed:** No.

### 4. Ingest AI assistant configuration files as first-class evidence sources

**Type:** enhancement

**Priority:** P1

**Proposed labels:** `type:enhancement`, `priority:p1`, `enhancement`

**Problem**

Repository truth increasingly lives in AI tool configuration files, but RepoNerve does not appear to treat the broader ecosystem of agent rules and steering files as a first-class source family.

**Competitive evidence**

Kiro uses `.kiro/steering/` for persistent project instructions. Cursor uses `.cursor/rules/*.mdc` and skills. CodeRabbit reads files such as `.cursorrules`, `CLAUDE.md`, and Copilot instructions for review context. Greptile uses repository rules. DeepWiki can be steered by `.devin/wiki.json`.

**RepoNerve alignment**

This strengthens knowledge preservation by indexing the instructions teams already write for agents. It is local-first and evidence-backed if stored as sources/facts with provenance.

**Acceptance criteria**

- Discover and ingest common agent configuration files, including:
  - `.cursor/rules/*.mdc`
  - `.cursor/skills/*/SKILL.md`
  - `CLAUDE.md`
  - `AGENTS.md`
  - `.github/copilot-instructions.md`
  - `.kiro/steering/*.md`
  - `.greptile/rules/**`
  - CodeRabbit configuration files
  - `.devin/wiki.json`
- Preserve exact file path and source type in evidence.
- Make indexed guidance visible through `ask`, `plan`, `review`, and MCP context packs.
- Add tests for deterministic ordering and duplicate handling.
- Do not execute or trust instructions as code; treat them as evidence-bearing documents.

**RFC needed:** Likely no if this extends document ingestion; yes if it introduces a new memory type or major ingestion model.

### 5. Expose fine-grained symbol navigation tools for MCP and CLI users

**Type:** feature

**Priority:** P1

**Proposed labels:** `type:feature`, `priority:p1`, `enhancement`

**Problem**

RepoNerve has deterministic code intelligence and explain commands, but agent users also need small, IDE-like symbol operations to avoid falling back to broad file reads or grep.

**Competitive evidence**

Serena markets MCP tools such as find symbol, symbol overview, references, declarations, implementations, and type hierarchy. Aider's repo map helps agents choose files through a symbol graph.

**RepoNerve alignment**

This fits the deterministic code intelligence layer and MCP-as-integration-surface strategy. It should expose existing indexed data through thin tools rather than adding LSP runtime dependencies.

**Acceptance criteria**

- Add CLI and MCP operations for a focused subset, such as:
  - file symbol outline
  - find symbol by qualified name
  - list callers and callees when available
  - list references or repository-code links for a symbol
  - dependency traversal for a file or symbol
- Every result includes source path, line range, and evidence.
- Results are deterministic and bounded by token budget.
- Add integration tests for Go and at least one tree-sitter language.
- Document when users should use these tools instead of `ask` or `explain`.

**RFC needed:** No if built on existing code stores and MCP patterns; yes if it changes MCP contracts incompatibly.

### 6. Generate a local source-cited repository guide from RepoNerve memory

**Type:** feature

**Priority:** P1

**Proposed labels:** `type:feature`, `priority:p1`, `enhancement`

**Problem**

RepoNerve can answer questions and serve an explore UI, but users also expect a browsable, source-cited onboarding artifact that captures the repository's architecture, features, ownership, and important decisions.

**Competitive evidence**

DeepWiki generates source-linked wiki pages, architecture diagrams, and repo Q&A for public repositories. Kiro creates requirements, design, and task documents before coding. These products reduce cold-start friction by creating visible artifacts, not only chat answers.

**RepoNerve alignment**

A deterministic local guide would package existing memory, code intelligence, feature understanding, and evidence links into a human-readable output without requiring cloud indexing or model-generated claims.

**Acceptance criteria**

- Add a command or documented workflow to export a local repository guide.
- Include sections for:
  - repository purpose
  - architecture overview
  - major features
  - important decisions and ADRs
  - ownership or expertise summary
  - suggested onboarding path
- Every conclusion links to repository evidence.
- Support Markdown output first; HTML can reuse the local `explore` UI later.
- Respect scoped scans and bounded output.

**RFC needed:** Possibly, if adding a new public command and product surface; not required for a documented composition of existing commands.

### 7. Add memory governance controls for session memory and handoff bundles

**Type:** enhancement

**Priority:** P1

**Proposed labels:** `type:enhancement`, `priority:p1`, `enhancement`

**Problem**

Local-first memory is a differentiator, but users need clear controls over what session memory stores, exports, forgets, and excludes.

**Competitive evidence**

Pieces emphasizes local processing plus granular controls such as app deny lists, pause modes, scoped deletion, and source visibility. This positions privacy controls as part of the memory product, not an afterthought.

**RepoNerve alignment**

RepoNerve already has `remember`, `forget`, and handoff bundles. Governance controls would strengthen trust while preserving local-first behavior.

**Acceptance criteria**

- Provide a way to list session memory entries by source, time, and type.
- Add scoped deletion or pruning for session memory beyond one-off forget operations.
- Add configurable deny patterns for files or paths that should never become session memory evidence.
- Add `doctor` checks or recommendations for memory freshness and sensitive-path risk.
- Ensure handoff exports include provenance and can be inspected before sharing.
- Add tests for deterministic filtering and deletion behavior.

**RFC needed:** Likely no for controls around existing session memory; yes if new memory types or storage migrations are required.

### 8. Improve first-run setup, PATH, MCP, and quickstart verification

**Type:** bug

**Priority:** P1

**Proposed labels:** `type:bug`, `priority:p1`, `bug`

**Problem**

First-run adoption depends on a tight feedback loop: install, initialize, scan, verify MCP, and ask the first useful question. If any step silently half-works, users may conclude the tool is broken.

**Competitive evidence**

Augment markets a two-minute MCP setup. DeepWiki starts from a GitHub URL. Kiro and Cursor surface project rules and steering setup inside the IDE. These products reduce setup ambiguity.

**RepoNerve alignment**

RepoNerve already has `init`, `scan`, `doctor`, `onboard`, npm distribution, and MCP configuration. The gap is packaging these into a verified success path.

**Acceptance criteria**

- After `init` or install, print a concise success checklist:
  - CLI binary available on PATH
  - memory database exists
  - scan completed or next scan command shown
  - MCP config written
  - IDE restart or MCP reload needed
  - first recommended commands
- Add a quick verification command or documented flow using `doctor`, `onboard`, and one sample `ask`.
- Detect common PATH problems and print shell-specific remediation.
- Avoid modifying generated integration files when memory already exists unless explicitly requested.

**RFC needed:** No.

### 9. Add context-pack diagnostics that explain why evidence was selected

**Type:** enhancement

**Priority:** P1

**Proposed labels:** `type:enhancement`, `priority:p1`, `enhancement`

**Problem**

RepoNerve emphasizes evidence-backed context, but users and agents need easier visibility into why a context pack included certain entities and omitted others.

**Competitive evidence**

Augment and CodeRabbit describe observability around context/tool use and review inputs. Aider documents its repo-map token budget and ranking behavior. Clear diagnostics build trust in retrieval systems.

**RepoNerve alignment**

This strengthens explainability and token economics without adding semantic search or model-based ranking.

**Acceptance criteria**

- Add an optional diagnostics mode for `ask`, `plan`, `review`, and MCP context tools.
- Report:
  - source services used
  - token or byte budget
  - number of candidate entities considered
  - selected entities with inclusion reasons
  - truncated or omitted sections
  - evidence source paths
- Keep diagnostics deterministic and machine-readable in JSON.
- Ensure compact/prose formats remain bounded.

**RFC needed:** No if optional and compatible; yes for breaking output-contract changes.

### 10. Document composition recipes with adjacent AI coding tools

**Type:** suggestion

**Priority:** P2

**Proposed labels:** `type:suggestion`, `priority:p2`, `documentation`

**Problem**

Many potential users already use Cursor, Claude Code, Aider, Serena, CodeRabbit, Greptile, Augment, Pieces, or DeepWiki. RepoNerve should explain how to compose with those tools instead of appearing to replace every adjacent category.

**Competitive evidence**

Augment positions Context Engine MCP as usable from Claude Code, Cursor, and Codex. Pieces exposes memory to coding assistants through MCP. DeepWiki exposes repository documentation through MCP. Serena integrates with many MCP clients.

**RepoNerve alignment**

RepoNerve is software understanding infrastructure. Composition recipes reinforce the "understanding layer" message and support growth without violating local-first or autonomy boundaries.

**Acceptance criteria**

- Add recipes for at least:
  - Cursor
  - Claude Code
  - Aider
  - Serena
  - CodeRabbit or Greptile
  - Pieces or DeepWiki
- Each recipe states:
  - what RepoNerve provides
  - what the adjacent tool provides
  - setup steps
  - recommended commands or MCP tools
  - privacy and evidence boundaries
- Include examples that avoid mandatory cloud dependencies for RepoNerve core.

**RFC needed:** No.
