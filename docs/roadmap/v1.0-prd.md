# Executive Summary

RepoNerve is a software understanding platform built around knowledge preservation. Its purpose is to ensure that software understanding survives beyond individual contributors and remains accessible to both humans and AI systems.

RepoNerve serves as a **software memory system**, preserving understanding that would otherwise be lost through contributor turnover, architectural evolution, and documentation drift.

The product chain:

```text
Knowledge Preservation
        ↓
Software Memory
        ↓
Software Understanding
```

Knowledge Preservation is the foundation. Software Memory is what organizations retain. Software Understanding is what humans and AI gain.

Previous milestones delivered Repository Intelligence as one capability within the knowledge preservation mission.

v1.0 completion requires Code Intelligence, Repository-Code Linking, Feature Understanding, and Development Experience (ISSUE-057) to deliver Software Understanding as the primary product outcome.

Completed capabilities:

* Repository Memory
* Repository Context
* Ownership Intelligence
* Knowledge Graph Intelligence
* Repository Intelligence
* Agent Context, Search, Sessions, and Workflows

These capabilities serve Knowledge Preservation and Repository Understanding.

v1.0 completion requires:

```text

Feature Understanding        ❌ Required — ISSUE-057
```

# Problem Statement

Software remembers code. Software forgets context.

Teams lose knowledge through turnover, evolution, documentation drift, and AI context limitations. Developers and AI agents repeatedly rediscover information that already existed.

Understanding software systems requires:

**Feature questions**

* What is the authentication feature?
* Who owns billing?
* What code implements search?
* What breaks if I change notifications?

**Understanding questions**

* What does this repository do?
* How does this work?
* Which files and functions are involved?
* What is the call graph?
* What depends on this symbol?

**Repository questions**

* Why does this exist?
* Who owns it?
* What breaks if I change it?
* Who should review this?

Without preserved, transferable knowledge connected to code, agents and developers repeatedly re-explore repositories and consume excessive tokens.

For most developers, `reponerve explain authentication` matters more than `reponerve explain-function CreateUser`.

# Repository-Code Linking

RepoNerve must support deterministic relationships between repository entities and code entities.

This is a core v1.0 requirement — not an implementation detail. Without Repository-Code Linking, Development Experience cannot combine repository and code context into unified Software Understanding.

## Required Link Types

* Decisions → code entities
* Facts → code entities
* Events → code entities
* Context → code entities

Relationship types include `DECISION_REFERENCES_CODE`, `FACT_REFERENCES_CODE`, and `EVENT_REFERENCES_CODE`.

## Example

```text
Decision:
ADR-004 OAuth Authentication

Linked Code:
  oauth.go
  AuthService (struct)
  LoginHandler (function)
```

This is what allows:

```bash
reponerve explain oauth
reponerve explain "authentication"
```

to return unified code context, repository context, and cross-authority links — not isolated retrieval from either intelligence layer alone.

## Requirements

* Deterministic link creation during ingestion and code indexing
* Evidence on every link
* Bidirectional traversal (repository entity → code entities, code entity → repository entities)
* Storage in `repository_code_relationships`

See: `docs/architecture/issue-057-architecture.md`

# Goals

## Goal 1 — Code Intelligence

Index code structure and build call graphs deterministically.

Success Criteria:

* Modules, packages, files, symbols, and endpoints are indexed
* Symbol resolution works (struct, interface, type_alias, function, method)
* Call graph and dependency analysis works
* Code Intelligence is authoritative for code understanding

---

## Goal 2 — Development Experience

Provide development-facing workflows that orchestrate all intelligence sources.

Success Criteria:

* `ask`, `explain`, `explain-file`, `explain-function`, `explain-struct`, `explain-interface`, `explain-type`, `plan`, `impact`, and `review` work end-to-end
* Explain output combines code context, repository context, and repository-code links
* No duplicate intelligence authorities

---

## Goal 3 — Agent Integration

Expose intelligence through MCP and structured context packages.

Success Criteria:

* Agents receive token-efficient, evidence-backed guidance
* MCP remains a thin interface over services
* Agent context includes repository-code links when available

---

## Goal 4 — Production Readiness

Operational readiness for v1.0 release after ISSUE-057 completion.

Success Criteria:

* Stable deployment model
* Documentation completeness
* All acceptance criteria pass

---

## Goal 5 — Feature Understanding

Humans think in features, not files.

Feature Understanding is a first-class v1.0 goal — not a secondary capability.

Success Criteria:

* Features can be explained through code and repository context
* Feature ownership can be identified
* Feature impact can be assessed
* Feature review guidance can be generated

Examples:

* Authentication
* Billing
* Search
* Notifications
* Metadata Management

Primary workflow:

```bash
reponerve explain authentication
```

resolves:

```text
Feature → Code → Ownership → Decisions → Impact
```

via Repository Search, Repository-Code links, Code Intelligence, and Repository Intelligence.

---

## Goal 6 — Knowledge Transfer

Make preserved software memory accessible without requiring direct access to original contributors.

Success Criteria:

* New contributors can understand repositories faster
* Ownership information remains accessible
* Architectural context remains discoverable
* Understanding survives contributor turnover
* AI systems can retrieve preserved knowledge without repeated repository exploration

Development Experience (`ask`, `explain`, `plan`, `impact`, `review`) is the primary knowledge transfer mechanism.

---

## Goal 7 — Token Intelligence (ISSUE-060)

Minimize LLM token consumption for software understanding.

Success Criteria:

* Graph-aware context compression with configurable token budgets
* `--format compact|prose|json` on CLI and MCP
* `reponerve hook install` and incremental scan on commit
* Measurable exploration token reduction per agent task

---

## Goal 8 — Evidence Graph & Session Memory (ISSUE-061)

Graph discovery and repository-scoped session memory.

Success Criteria:

* Community detection and surprising connections on knowledge graph
* `reponerve explore` with HTML export
* `remember` / `forget` with evidence provenance
* Agent handoff bundles and fixed workflow templates

---

## Goal 9 — Multi-Language Code Intelligence (ISSUE-062)

Extend code indexing beyond Go.

Success Criteria:

* Tree-sitter indexing for TypeScript, Python, Rust
* Incremental per-file updates
* Repository-code linking works across indexed languages

---

## Goal 10 — Foundation Fixes (ISSUE-059)

Ship quick wins on Repository Intelligence before and during ISSUE-057.

Success Criteria:

* Expertise detection runs during scan
* Agent search and graph impact exposed via CLI
* Ingestion debt resolved (scan help text, FTS5)

# Non-Goals

v1.0 does not introduce. **Agreed exclusions:**

| Non-Goal | Rationale |
| --- | --- |
| Semantic / hybrid embedding search | Deterministic, evidence-backed retrieval only |
| User-defined workflow composition | Fixed workflow templates only (ISSUE-061) |
| Autonomous code modification or deployment | Understanding infrastructure, not an coding agent |
| Cloud-required core product | Local-first; no mandatory SaaS |
| Cross-repo enterprise federation | Single-repository depth for v1.0 |

Additional non-goals:

* LLM-required parsing or routing
* Duplicate intelligence scoring systems

RepoNerve remains deterministic intelligence infrastructure.

See `docs/roadmap/v1.x-backlog.md`.

# Architecture Principles

## Dual Authority Model

* Code Intelligence is authoritative for code understanding
* Repository Intelligence is authoritative for repository knowledge
* Development Experience orchestrates both
* Repository-Code Linking connects both authorities deterministically

---

## Reuse Before Reinvention

Development Experience must reuse:

* Repository Search
* Ownership Intelligence
* Knowledge Graph
* Discovery, Learning, Reviewers, Change Planning
* Workflow Intelligence
* Agent Context Builder

---

## Evidence Preservation

All outputs must preserve evidence, explanations, and provenance.

Evidence-free conclusions are invalid.

---

## Determinism

The same repository state must produce identical outputs.

# Success Definition

Software Understanding is the outcome. A developer or AI agent should be able to understand, change, review, and evolve software without repeatedly rediscovering repository context.

Concrete workflows:

* Explain this feature (`reponerve explain authentication`)
* Explain this code (`reponerve explain-file`, `reponerve explain-function`)
* Identify who owns an area
* Understand what breaks if they change something
* Know who should review a change
* Plan work such as "Add OAuth login"
* Onboard to an unfamiliar repository with preserved knowledge

using RepoNerve alone.

Knowledge survives contributor turnover. Understanding is transferable. Software memory persists.
